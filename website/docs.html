Quick Start

Migrate a GitLab CI pipeline in three steps:

Terminal 
# 1. Convert your pipeline
gl2gh migrate .gitlab-ci.yml

# 2. Review the generated workflow
cat .github/workflows/ci.yml

# 3. Validate the output
gl2gh validate .github/workflows/
ℹ
By default, migrate runs validation and optimization checks after conversion. Use --pure-rules to skip those checks, or --ai for full AI-enhanced migration.

CLI: migrate

The primary command. Reads a .gitlab-ci.yml file and produces GitHub Actions workflow files.

Usage 
gl2gh migrate [OPTIONS] [INPUT_FILE]
Option Default Description
INPUT_FILE .gitlab-ci.yml Path to the GitLab CI pipeline file
-o, --output-dir .github/workflows Directory for generated workflow files
-n, --name CI Name of the generated workflow
--ai off Use GitHub Copilot AI for enhanced conversion
--pure-rules off Skip validation & optimization (pure rule-based)
--model gpt-4.1 AI model when using --ai
--dry-run off Print output without writing files
-v, --verbose off Show detailed conversion info
--format yaml Output format: yaml or json

Examples

Terminal 
# Basic conversion with validation + optimization
gl2gh migrate .gitlab-ci.yml

# AI-enhanced conversion
gl2gh migrate .gitlab-ci.yml --ai

# Pure rule-based (no agent checks)
gl2gh migrate .gitlab-ci.yml --pure-rules

# Preview without writing files
gl2gh migrate .gitlab-ci.yml --dry-run --verbose

# Custom workflow name and output
gl2gh migrate .gitlab-ci.yml -n "Build & Deploy" -o workflows/

CLI: inspect

Analyze a GitLab CI pipeline without converting it. Shows stages, jobs, variables, templates, and includes.

Terminal 
gl2gh inspect .gitlab-ci.yml
gl2gh inspect .gitlab-ci.yml --verbose   # includes per-job details

CLI: validate

Validate generated GitHub Actions YAML files for syntax correctness.

Terminal 
gl2gh validate .github/workflows/

CLI: migrate-repo

Full repository migration from GitLab to GitHub. Uses gh CLI for GitHub operations when available.

Terminal 
gl2gh migrate-repo git@gitlab.com:org/repo.git org/new-repo
gl2gh migrate-repo git@gitlab.com:org/repo.git org/new-repo --ai
Option Default Description
SOURCE_GITLAB_REPO — GitLab repository URL or path
TARGET_GITHUB_REPO — Target GitHub repository
--branch main Branch to push
--ai off Use AI-enhanced migration

CLI: gh-status

Check gh CLI and GitHub Copilot CLI availability, and list active workflows.

Terminal 
gl2gh gh-status

How It Works

gl2gh uses a hybrid approach: deterministic rule-based translation for common CI constructs, and AI-powered agents for complex patterns.

Parse YAML → Build Models → Rule-Based Convert → Validate → Optimize → Output YAML
  1. Parse — GitLabCIParser reads the .gitlab-ci.yml using ruamel.yaml (preserves anchors & aliases). Resolves extends templates at parse time.
  2. Model — Parsed data is mapped into typed Python dataclasses (GitLabPipeline, GitLabJob, etc.) as an intermediate representation.
  3. Convert — GitLabToGitHubConverter applies 30+ pure mapping rules to translate variables, stages, cache, artifacts, triggers, and more.
  4. Validate — ValidatorAgent runs static checks — YAML syntax, required keys, job structure, and 9 security injection pattern detections.
  5. Optimize — OptimizerAgent scores the workflow (0–100) and suggests improvements for caching, parallelism, concurrency, and timeouts.
  6. Output — Final YAML is written to .github/workflows/ with a source-tracking header comment.

AI-Enhanced Mode

When --ai is passed, gl2gh uses GitHub Copilot (via the Chat API) to handle complex patterns that lack direct one-to-one mappings:

  • Multi-project pipelines
  • Advanced rules: conditions with complex logic
  • Custom runner configurations
  • Complex extends chains
  • Dynamic child pipelines
Rule-Based Baseline → Send to Copilot → Tool-Use Loop (up to 8 iterations) → Enhanced Result
⚠
Requires GITHUB_TOKEN with Copilot access. If AI enhancement fails, the tool falls back to the rule-based result with a warning.
Terminal 
export GITHUB_TOKEN=ghp_...
gl2gh migrate .gitlab-ci.yml --ai --verbose

Validator Agent

The Validator Agent runs automatically on every conversion (unless --pure-rules is set). It performs static analysis on the generated workflow:

Check Severity Description
YAML syntaxErrorEnsures valid YAML output
Required keysErrorChecks for name, on, jobs
Job structureErrorEach job must have runs-on and steps
Caller jobsOKReusable workflow callers (uses:) skip runs-on check
Security: expression injectionWarningDetects 9 known injection patterns (e.g. github.event.issue.title)
Non-standard runnersInfoFlags custom runner labels

Optimizer Agent

The Optimizer Agent scores your workflow and suggests improvements:

Category Points What It Checks
Baseline50Starting score for any valid workflow
Caching+10Presence of actions/cache or setup-* with cache
Concurrency+10concurrency key to cancel redundant runs
Parallelism+10Multiple independent jobs that can run in parallel
Checkout+5actions/checkout@v4 present in every job
Timeout+5timeout-minutes set on jobs
Matrix builds+5strategy.matrix for multi-version testing

Quality Score

After conversion, gl2gh displays a quality score from 0 to 100:

Score Rating Action
70–100GoodWorkflow is well-optimized
50–69FairConsider applying suggested optimizations
0–49Needs workTry --ai mode for better results
✔
If the score falls below 60, gl2gh automatically suggests using --ai mode in the conversion notes.

Web API
