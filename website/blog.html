Building a GitLab CI to GitHub Actions Migration Tool

How I built gl2gh — a hybrid rule-based and AI-powered CLI tool that converts .gitlab-ci.yml pipelines into GitHub Actions workflows, handling 30+ variable translations, stage dependency graphs, and complex CI patterns.

8 min read

The Problem

If you've ever had to migrate a project from GitLab to GitHub, you know the CI/CD pipeline is one of the hardest parts to bring over. The Git repository itself is straightforward — add a remote and push. But the pipeline? That's a different story.

GitLab CI and GitHub Actions have fundamentally different models. GitLab uses sequential stages where jobs automatically depend on the previous stage. GitHub Actions uses an explicit dependency graph with the needs keyword. Variables have different names, caching works through dedicated actions rather than built-in keywords, and concepts like extends templates and only/except rules have no direct GitHub equivalent.

I found myself repeatedly doing this conversion by hand across projects — reading GitLab CI docs, cross-referencing with Actions docs, translating variables one by one, restructuring stages into dependency graphs. Every pipeline took an hour or more of tedious, error-prone work. So I decided to automate it.

Existing Solutions

Before writing any code, I looked for existing tools. GitHub has an official migration guide but it's a reference document, not an automation tool. It gives you the concept mappings but leaves the actual translation as manual work.

There are a few community scripts and converters, but most handle only the most basic cases — simple script steps and image declarations. None of them handle the full breadth of GitLab CI features:

  • Variable translation ($CI_COMMIT_SHA → ${{ github.sha }}, 30+ mappings)
  • Stage-based ordering to explicit needs dependency graphs
  • extends and YAML anchors for template resolution
  • Cache and artifact conversion to dedicated Actions
  • Docker-in-Docker builds with service containers
  • Matrix builds (parallel.matrix → strategy.matrix)
  • rules: conditions with variable-aware if: expressions
  • Environment and deployment configurations
  • Child/nested pipelines with trigger:

I didn't find anything that covered even half of these, so I started building gl2gh.

The Approach

The core insight was that roughly 80% of CI migration work is deterministic. Variable names have known translations. Stages map to dependency graphs algorithmically. Cache keys follow predictable patterns. These don't need AI — they need a lookup table and some careful conversion logic.

The remaining 20% — multi-project pipelines, advanced rules: conditions, custom runner configurations, complex Docker compositions — often lack a clean one-to-one mapping. These benefit from an AI agent that can reason about intent and generate appropriate GitHub Actions equivalents.

Parsing: Handling YAML Anchors and Extends

GitLab CI heavily uses YAML anchors (&anchor) and aliases (*anchor) for template reuse, plus the extends keyword for job inheritance. Standard YAML parsers like PyYAML flatten anchors on load, losing the template structure.

I chose ruamel.yaml specifically because it preserves anchor/alias relationships. The parser resolves extends at parse time by merging template properties into jobs, so the converter only ever sees fully resolved job definitions. Template jobs (prefixed with ., like .python_base) are flagged as is_template=True and excluded from the final workflow.

Conversion: Pure Functions and Lookup Tables

The conversion engine is built entirely on pure functions — no side effects, no I/O. Every mapping rule takes a GitLab value in and returns a GitHub Actions equivalent out. This makes them independently testable and easy to extend.

Here's a sample of the variable translation table:

GitLab CI Variable GitHub Actions Equivalent
$CI_COMMIT_SHA ${{ github.sha }}
$CI_COMMIT_REF_NAME ${{ github.ref_name }}
$CI_PIPELINE_ID ${{ github.run_id }}
$CI_REGISTRY_IMAGE ghcr.io/${{ github.repository }}
$CI_REGISTRY_PASSWORD ${{ secrets.GITHUB_TOKEN }}
$CI_MERGE_REQUEST_IID ${{ github.event.pull_request.number }}
$CI_PROJECT_DIR ${{ github.workspace }}

Stage ordering is converted algorithmically. The stages_to_needs_graph function takes the ordered list of stages and builds an explicit dependency map: every job in stage N depends on all jobs in stage N-1. This preserves the sequential guarantee while giving GitHub Actions the explicit graph it requires.

AI Enhancement: The Copilot Agent

For complex patterns that the rule engine can't handle cleanly, gl2gh optionally sends the baseline output to a GitHub Copilot agent with tool definitions. The agent enters an agentic loop — Copilot emits tool calls (validate_yaml, save_workflow, add_warning), the agent executes them and feeds results back, repeating for up to 8 iterations until the model is satisfied.

Design decision

AI enhancement is strictly opt-in via the --ai flag. The rule-based converter always produces output first. If AI enhancement fails for any reason, the system falls back to the rule-based result with a warning. The user always gets usable output.

Validation and Scoring

Every converted workflow is automatically run through a ValidatorAgent (YAML syntax, required keys, job structure, 9 security injection patterns) and an OptimizerAgent that scores the workflow from 0 to 100. The score considers caching, concurrency groups, parallelism, timeouts, and other best practices. If the score falls below 60, gl2gh suggests using --ai mode for better results.

Before and After

Here's a real example. A typical Node.js GitLab CI pipeline with 4 stages, caching, artifacts, and a deploy job that only runs on main:

.gitlab-ci.yml Input 
stages:
  - install
  - test
  - build
  - deploy

variables:
  NODE_VERSION: "20"

install:
  stage: install
  image: node:20
  script:
    - npm ci
  cache:
    key: ${CI_COMMIT_REF_SLUG}
    paths:
      - node_modules/

test:
  stage: test
  image: node:20
  script:
    - npm test
  artifacts:
    reports:
      junit: test-results.xml

build:
  stage: build
  image: node:20
  script:
    - npm run build
  artifacts:
    paths:
      - dist/
    expire_in: 1 week

deploy:
  stage: deploy
  image: node:20
  script:
    - npm run deploy
  environment:
    name: production
  only:
    - main
github-actions.yml Output 
name: CI
on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  install:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/cache@v4
        with:
          path: node_modules/
          key: ${{ github.ref_name }}
      - run: npm ci

  test:
    runs-on: ubuntu-latest
    needs: [install]
    steps:
      - uses: actions/checkout@v4
      - run: npm test
      - uses: dorny/test-reporter@v1
        with:
          name: Tests
          path: test-results.xml
          reporter: java-junit

  build:
    runs-on: ubuntu-latest
    needs: [test]
    steps:
      - uses: actions/checkout@v4
      - run: npm run build
      - uses: actions/upload-artifact@v4
        with:
          name: build-artifacts
          path: dist/
          retention-days: 7

  deploy:
    runs-on: ubuntu-latest
    needs: [build]
    environment: production
    steps:
      - uses: actions/checkout@v4
      - run: npm run deploy

Note how stages became needs dependencies, the cache became an actions/cache@v4 step, artifacts became actions/upload-artifact@v4, the JUnit report became dorny/test-reporter@v1, and only: [main] became the on: push trigger filter.

Key Design Decisions

📦 Dataclass Models
GitLabJob, GitLabPipeline, and ConversionResult are plain Python dataclasses — not Pydantic. They provide typed intermediate representations between parsing and conversion without the overhead of validation frameworks.
🔧 Pure Mapping Functions
All functions in mappings/rules.py are pure — no side effects, no I/O. This makes them trivially testable and allowed me to build up 126 tests with confidence.
🌐 ruamel.yaml over PyYAML
GitLab CI's heavy use of YAML anchors and aliases made ruamel.yaml essential. PyYAML is only used for simpler validation tasks where anchor preservation isn't needed.
🛡️ Graceful AI Fallback
If AI enhancement fails (network error, rate limit, bad response), the system falls back to the rule-based result automatically. The user always gets usable output.
📁 Local-First Output
Workflows are written to disk for review, never pushed directly to GitHub. The gh CLI integration is optional convenience, not a requirement.
🔍 Template Resolution at Parse Time
extends is resolved in the parser, not the converter. The converter only sees fully resolved jobs, which simplifies the conversion logic significantly.

How to Use

Install gl2gh (requires Python 3.11+):

pip install -e ".[dev]"

Convert a pipeline:

# Basic conversion
gl2gh migrate .gitlab-ci.yml

# Preview without writing files
gl2gh migrate .gitlab-ci.yml --dry-run

# With AI enhancement
export GITHUB_TOKEN="ghp_..."
gl2gh migrate .gitlab-ci.yml --ai

# Pure rule-based (skip validation/optimization)
gl2gh migrate .gitlab-ci.yml --pure-rules

# Inspect pipeline structure
gl2gh inspect .gitlab-ci.yml -v

# Validate generated workflows
gl2gh validate .github/workflows

# Full repo migration
gl2gh migrate-repo git@gitlab.com:org/repo.git org/repo --ai

Or use the web converter — paste your YAML and get the result instantly.

What I Learned

Building gl2gh reinforced a few things I've come to believe about developer tooling:

  1. Start with deterministic rules, add AI later. The rule-based converter handles the vast majority of real-world pipelines. AI is a force multiplier for edge cases, not a replacement for understanding the domain.
  2. Pure functions make testing painless. With 126 tests and zero I/O in the core conversion path, I can refactor aggressively without fear. Every mapping rule has its own test case.
  3. YAML is harder than it looks. Anchors, aliases, merge keys, and the ambiguity of "is this a job or a global keyword?" made the parser the most complex piece. The GLOBAL_KEYWORDS set in the parser determines which root keys are configuration vs. job definitions.
  4. Always produce output. Even if the conversion is imperfect, giving the user a 90% correct workflow that they can manually fix is vastly better than failing with an error. Warnings and conversion notes guide them to the remaining manual work.
